## 【一、MVP范围定义】

### 1.1 核心边界

学段范围：
- MVP包含：小学（1-6年级）、初中（7-9年级）
- 排除范围：幼儿园、高中

学科范围：
- MVP包含：语文、数学、英语
- 排除范围：物理、化学、生物、历史、地理、政治

教学环节：
- MVP包含：备课（教案生成）、练习（出题+批改）
- 排除范围：授课、深度分析、习惯养成

习作批改：
- 实现方式：全自动（拍照→识别→批改→教师复核）
- 备注：卷面字迹评估已纳入（无额外成本）

教师端：PC Web，不含手机端教师功能

学生端：手机小程序（含拍照上传），不含PC端、平板App

家长端：只读查看 + 签字确认，不含多孩切换、打印、习惯监督

学校端：基础配置（账号、班级），不含深度报表、教研管理

### 1.2 不进入MVP的功能清单

1. 高中/幼儿园支持 - 原因：学段差异大，需独立建模
2. 物理/化学/生物/史地政 - 原因：学科逻辑复杂，需单独设计
3. 电子白板授课 - 原因：需与硬件适配
4. 习惯养成 - 原因：需独立的产品逻辑
5. 假期分时段管控 - 原因：场景复杂度高
6. 多孩认证与切换 - 原因：产品逻辑需重构
7. 解答题自动批改 - 原因：技术成熟度不足（但作文批改已纳入）
8. AI分身/替身 - 原因：不对外宣传，仅内部优化
9. 教育局审计接口 - 原因：二期预留

### 1.3 MVP核心原则

原则一：全自动化
拒绝一切手工录入，能用技术解决的绝不推给教师

原则二：技术天花板不设限
架构支持模型热替换，随大模型版本迭代自然进化

原则三：教师复核兜底
低置信度内容推送教师确认，不承诺100%准确

原则四：课标对齐为底线
所有教案必须可溯源到课标条目

原则五：小微即实习生
教师最终确认，不给惊喜也不给惊吓


## 【二、功能清单】

### 2.1 P0功能（必须，MVP核心）

【教师端 - PC Web】

1. 账号登录
   说明：手机号+验证码
   小微角色：无

2. 班级选择
   说明：切换任教班级
   小微角色：无

3. 教案生成
   说明：输入课题→小微生成结构化教案
   小微角色：执行者

4. 教案编辑
   说明：富文本编辑器
   小微角色：辅助

5. 教案定稿
   说明：确认后进入教案库
   小微角色：无

6. 课标对齐校验
   说明：自动标注教案对应的课标条目
   小微角色：执行者

7. 基础出题
   说明：根据知识点生成5-10道题
   小微角色：执行者

8. 作业发布
   说明：选择班级、设置截止时间
   小微角色：无

9. 中文作文批改
   说明：查看学生拍照→小微识别+预打分→教师复核
   小微角色：执行者+汇报

10. 英文写作游戏化
    说明：查看学生填空式/选择式写作→小微反馈
    小微角色：执行者

11. 确定性题目批改
    说明：自动批改（选择+填空+计算）
    小微角色：执行者

12. 低置信度复核队列
    说明：小微推送需要教师确认的批改项
    小微角色：主动汇报

13. 小微对话窗
    说明：浮窗式AI助手，可自然语言交互
    小微角色：核心交互

【学生端 - 手机小程序】

1. 账号登录 - 家长/学校导入的账号
2. 作业列表 - 查看待做、已做作业
3. 客观题答题 - 选择/填空/计算题输入
4. 作文拍照上传 - 调用相机拍照，自动上传
5. 英文写作游戏 - 填空式/选择式/仿写式完成
6. 提交 - 一键提交
7. 查看批改 - 查看分数+解析+修改建议

【家长端 - 手机小程序】

1. 账号登录 - 绑定孩子账号
2. 作业查看 - 查看孩子作业完成情况（只读）
3. 作业确认签字 - 点击“已阅”+手写签名
4. 通知接收 - 接收作业提醒、签字提醒

【学校管理端 - PC Web】

1. 学校注册 - 填写学校信息
2. 账号管理 - 创建教师、学生账号
3. 班级配置 - 设置年级、班级、任课关系
4. 基础报表 - 查看教案数量、作业完成率

### 2.2 小微交互设计

【位置与形态】

PC Web：右下角浮窗，可折叠、可拖拽
小程序：底部导航栏/悬浮球，轻量级

【核心对话场景】

场景1：教师说“帮我写一份《分数》教案”
小微行为：生成初稿

场景2：教师说“把导入环节改成游戏”
小微行为：局部修改

场景3：教师说“出10道计算题”
小微行为：生成题目

场景4：教师说“这几个学生作文帮我看看”
小微行为：批量预打分

场景5：教师说“这个扣分标准记住了”
小微行为：学习教师偏好

场景6：教师说“今天有什么待办？”
小微行为：汇报作业待批改

【小微人格设定】

角色：助教/实习生
语气：尊重、积极、略青涩但不怯场
犯错时：承认错误，记录学习
被纠正时：说“明白了，下次我会注意”
主动汇报：说“老师，这3篇作文我把握不大，请您看看”


## 【三、模型体系规划】

### 3.1 模型清单

M0：主大模型
功能：通用理解、生成基础
MVP阶段策略：调用DeepSeek/Qwen API
独立训练必要性：不训练

M1：手写识别与卷面评估
功能：识别手写作文，输出字迹评分
MVP阶段策略：多模态API + 置信度提取
独立训练必要性：二期微调

M2：课标对齐与学科逻辑
功能：判断教案是否符合课标
MVP阶段策略：专家知识图谱 + 规则引擎
独立训练必要性：必要

M3：教师风格蒸馏
功能：学习教师个人偏好
MVP阶段策略：RAG检索，不单独训练
独立训练必要性：二期轻量微调

M4：批改评分模型
功能：对主观题评分
MVP阶段策略：主模型预打分 + 教师复核
独立训练必要性：必要（持续收集数据）

M5：出题模型
功能：生成符合难度的题目
MVP阶段策略：主模型 + Prompt
独立训练必要性：可合并

M6：互动脚本模型
功能：生成课堂互动
MVP阶段策略：主模型 + 模板库
独立训练必要性：可合并

### 3.2 卷面字迹评估的技术创新

指标一：整体潦草度
计算方式：1 - 平均置信度
用途：卷面分主要依据

指标二：涂改频率
计算方式：涂改区域数 / 总字数
用途：卷面整洁度

指标三：低置信度区域分布
计算方式：聚类分析
用途：是否局部潦草 vs 整体潦草

归因逻辑：用识别置信度分布自动区分“字迹问题”和“内容问题”，无需额外模型。

### 3.3 模型训练数据来源

D1：教师批改行为
D2：教师修改教案行为
D3：教师出题偏好
D4：专家标注数据
D5：学生答题数据

各模型的数据依赖关系：
- M1（手写识别）依赖 D1
- M2（课标对齐）依赖 D4
- M3（风格蒸馏）依赖 D2
- M4（批改评分）依赖 D1 + D5
- M5（出题）依赖 D3 + D5


## 【四、技术架构】

### 4.1 服务清单

user-service：用户与权限管理，使用Go语言
lesson-service：教案CRUD、生成，使用Python语言
homework-service：作业考试管理，使用Go语言
grading-service：批改服务，使用Python语言
home-school-service：家校服务，使用Go语言
xiaowei-service：小微对话服务，使用Python语言
notification-service：推送服务，使用Node.js语言
ai-gateway：AI能力网关，使用Python语言

### 4.2 技术栈

前端PC：React + ProseMirror
前端小程序：微信小程序原生
后端：Go + Python FastAPI
数据库：MySQL 8.0 + 向量数据库
缓存：Redis
消息队列：Kafka
对象存储：阿里云OSS / MinIO
容器化：Docker + K8s

### 4.3 部署模式

SaaS多租户：一校一实例，逻辑隔离
数据隔离：每校独立database/schema，Redis key前缀{school_id}
私有化部署：大校/教育局可选


## 【五、数据指标定义】

### 5.1 MVP成功指标

指标1：教案生成时间
目标值：≤30秒（P99）
测量方式：系统日志

指标2：教案接受率
目标值：≥65%（教师修改<20%即接受）
测量方式：日志分析

指标3：教案-课标对齐率
目标值：100%
测量方式：每份都有课标标注

指标4：出题可用率
目标值：≥80%
测量方式：教师评分≥4/5

指标5：批改准确率（客观题）
目标值：≥95%
测量方式：与标准答案对比

指标6：作文识别教师可接受率
目标值：≥80%
测量方式：教师复核确认率

指标7：作业提交率
目标值：≥70%
测量方式：系统统计

指标8：小微对话满意度
目标值：≥4/5
测量方式：教师评价

### 5.2 小微专项指标

指标1：自然语言意图识别准确率，目标值≥85%
指标2：教师主动调用小微频率，目标值日均≥5次/教师
指标3：教师纠正小微的响应率，目标值≥70%
指标4：小微推荐采纳率，目标值≥60%

## 【八、风险与应对】

风险1：多模态识别准确率不足
应对措施：低置信度推教师复核；架构支持模型热替换

风险2：课标对齐规则不完整
应对措施：专家持续标注；规则可配置

风险3：小微理解偏差
应对措施：教师可纠正；记录差异用于训练

风险4：教师接受率低
应对措施：试点学校深度参与；简化操作

风险5：开发进度延期
应对措施：功能极致收敛；每周进度检查
